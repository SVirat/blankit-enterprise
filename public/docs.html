Blankit API Documentation

The Blankit API provides enterprise-grade PII and PHI detection and redaction as a simple REST service. Send text or upload documents — receive clean, redacted output plus a reversible redaction map.

Zero data retention. Blankit processes all data in memory. Nothing is logged, stored, or cached on our servers after the response is returned.

Authentication

All API requests require a Bearer token in the Authorization header:

HTTP Header
Authorization: Bearer Blankit_live_abc123def456

API keys are prefixed with blankit_live_ for production and Blankit_test_ for sandbox. Keep your keys secret — do not expose them in client-side code.

Key prefixEnvironmentBilling
blankit_test_SandboxFree (limited to 100 req/day)
blankit_live_ProductionPer your plan

Base URL

https://api.Blankit.dev/v1

All endpoints are versioned. The current version is v1. We will never introduce breaking changes without incrementing the version.

Error Handling

The API uses standard HTTP status codes. All error responses have a consistent JSON shape:

Error Response
{
  "error": {
    "code": "invalid_api_key",
    "message": "The provided API key is invalid or expired.",
    "status": 401
  }
}
StatusCodeDescription
400bad_requestMissing or malformed request body
401invalid_api_keyMissing, invalid, or expired API key
413payload_too_largeText exceeds 500KB or document exceeds 50MB
422unsupported_formatUploaded file type is not supported
429rate_limitedToo many requests — slow down

POST

/v1/redact

Redact PII from a plain text string. Returns the redacted text and a map of all detected items.

Request Body

FieldTypeRequiredDescription
textstringYesThe input text to redact (max 500KB).
categoriesobjectNoOverride default detection categories. Each key is a category name (emails, phones, ssn, creditCards, addresses, names, dates, medical, ip) mapped to true/false.
custom_wordsstring[]NoAdditional organization-specific words to redact.

Example Request

cURL
curl -X POST https://api.Blankit.dev/v1/redact \
  -H "Authorization: Bearer blankit_live_abc123" \
  -H "Content-Type: application/json" \
  -d '{
    "text": "Please update the file for John Smith. His SSN is 123-45-6789 and email is john@example.com.",
    "categories": {
      "emails": true,
      "ssn": true,
      "names": true
    }
  }'

Example Response

200 OK
{
  "redacted_text": "Please update the file for [NAME_1]. His SSN is [SSN_2] and email is [EMAIL_3].",
  "items": [
    { "type": "Person Name", "placeholder": "[NAME_1]", "index": 0 },
    { "type": "SSN", "placeholder": "[SSN_2]", "index": 1 },
    { "type": "Email", "placeholder": "[EMAIL_3]", "index": 2 }
  ],
  "redaction_map": {
    "[NAME_1]": "John Smith",
    "[SSN_2]": "123-45-6789",
    "[EMAIL_3]": "john@example.com"
  },
  "stats": {
    "total_redactions": 3,
    "categories_matched": ["Person Name", "SSN", "Email"],
    "input_tokens": 19,
    "processing_ms": 12
  }
}
Security note: The redaction_map contains the original sensitive data. Store it securely on your server or client — never forward it to an LLM.
POST

/v1/redact-doc

Upload a document for redaction. Supported formats: DOCX, XLSX, PPTX, PDF, TXT, CSV, JSON, XML, MD, HTML, YAML, LOG, INI, RTF.

Request

Send as multipart/form-data.

FieldTypeRequiredDescription
filefileYesThe document to redact (max 50MB).
categoriesJSON stringNoCategory overrides (same shape as text endpoint).
custom_wordsJSON stringNoJSON array of custom words to redact.

Example Request

cURL
curl -X POST https://api.Blankit.dev/v1/redact-doc \
  -H "Authorization: Bearer blankit_live_abc123" \
  -F "file=@report.docx" \
  -F 'categories={"emails":true,"ssn":true,"names":true}'

Example Response

200 OK
{

POST

/v1/unredact

Restore original values in an LLM response using a previously returned redaction map. Call this on your backend — never expose original PII to the LLM.

Request Body

FieldTypeRequiredDescription
textstringYesThe LLM response containing placeholders like [NAME_1].
redaction_mapobjectYesThe map returned by the original /v1/redact or /v1/redact-doc call.

Example

cURL
curl -X POST https://api.Blankit.dev/v1/unredact \
  -H "Authorization: Bearer blankit_live_abc123" \
  -H "Content-Type: application/json" \
  -d '{
    "text": "Based on the records for [NAME_1], I recommend...",
    "redaction_map": {
      "[NAME_1]": "John Smith"
    }
  }'

Response

{
  "restored_text": "Based on the records for John Smith, I recommend...",
  "replacements_made": 1
}

Quickstart Guide

Get up and running in under 5 minutes.

1. Get your API key

Contact us to receive your API key. You'll get both a sandbox key (blankit_test_) and a production key (blankit_live_).

2. Install the SDK (optional)

npm
npm install @Blankit/sdk

Or call the REST API directly from any language.

3. Redact before sending to your LLM

Node.js
const blankit_KEY = process.env.blankit_API_KEY;

// Step 1: Redact PII from user prompt
const redactRes = await fetch('https://api.Blankit.dev/v1/redact', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${Blankit_KEY}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    text: userPrompt,
    custom_words: ['ProjectNova', 'AcmeCorp']
  })
});

const { redacted_text, redaction_map } = await redactRes.json();

// Step 2: Send CLEAN text to OpenAI (no PII)
const llmRes = await openai.chat.completions.create({
  model: 'gpt-4',
  messages: [{ role: 'user', content: redacted_text }]
});

const llmOutput = llmRes.choices[0].message.content;

// Step 3: Restore original values for the user
const restoreRes = await fetch('https://api.Blankit.dev/v1/unredact', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${blankit_KEY}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    text: llmOutput,
    redaction_map
  })
});

const { restored_text } = await restoreRes.json();
// Show restored_text to the user — LLM never saw real PII ✓

4. Python example

Python
import requests, os

API_KEY = os.environ["blankit_API_KEY"]
BASE    = "https://api.Blankit.dev/v1"
HEADERS = {"Authorization": f"Bearer {API_KEY}", "Content-Type": "application/json"}

# Redact
resp = requests.post(f"{BASE}/redact", headers=HEADERS, json={
    "text": "Contact Jane Doe at jane@acme.com or 555-123-4567"
})
data = resp.json()

print(data["redacted_text"])
# "Contact [NAME_1] at [EMAIL_2] or [PHONE_3]"

print(data["redaction_map"])
# {"[NAME_1]": "Jane Doe", "[EMAIL_2]": "jane@acme.com", "[PHONE_3]": "555-123-4567"}

Custom Dictionaries

Beyond the 9 built-in categories, you can supply a custom_words array with every request. These words will be redacted with [CUSTOM_N] placeholders.

{
  "text": "The ProjectNova launch is scheduled for Q3. Contact the Titan team.",
  "custom_words": ["ProjectNova", "Titan"]
}

Response:

{
  "redacted_text": "The [CUSTOM_1] launch is scheduled for Q3. Contact the [CUSTOM_2] team.",
  "redaction_map": {
    "[CUSTOM_1]": "ProjectNova",
    "[CUSTOM_2]": "Titan"
  }
}
Tip: For large dictionaries (100+ terms), consider maintaining them server-side and sending them via the custom_words parameter. Contact us about persistent org-level dictionaries on Enterprise plans.

LLM Integration Pattern

The recommended integration architecture for protecting user data when using any LLM:

User Input
"Update John Smith's record, SSN 123-45-6789"
→
Blankit API
POST /v1/redact → receives clean text + map
→
LLM (GPT-4, Claude, etc.)
"Update [NAME_1]'s record, SSN [SSN_2]"
→
Blankit API
POST /v1/unredact → restores original values
→
User sees response
With real names, SSNs, etc. restored
Important: Never forward the redaction_map to the LLM. Store it temporarily on your backend and use it only for the unredact step.
